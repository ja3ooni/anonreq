# بنية بوابة AnonReq

توفر هذه الوثيقة ملخصًا لبنية بوابة AnonReq. وهي توضح تفاصيل طوبولوجيا المكونات ودورة حياة الطلب وضمانات الأمان الأساسية.

## نظرة عامة على النظام

بوابة AnonReq هي بوابة أمان وأرشفة ذكاء اصطناعي مستضافة ذاتيًا مصممة لتقع بين تطبيقات المؤسسة وموفري LLM (مادلات اللغة الكبيرة) الخارجيين أو المحليين. من خلال العمل كوكيل اعتراض، فإنها تضمن تصنيف البيانات الحساسة (مثل PII و PHI و PCI أو الأسرار التجارية) وتطهيرها واستبدالها بالرموز قبل عبور حد الثقة لشبكة المؤسسة.

```
┌─────────────────┐       ┌─────────────────┐       ┌───────────────────┐
│                 │       │                 │       │                   │
│      تطبيق      │──────>│     بوابة       │──────>│     موفرو LLM     │
│    المؤسسة      │<──────│    AnonReq      │<──────│ (OpenAI, Gemini)  │
│                 │       │ (مستضافة ذاتيًا) │       │                   │
└─────────────────┘       └─────────────────┘       └───────────────────┘
```

## دورة حياة الطلب

يمر كل طلب يتم إرساله إلى موفر خارجي عبر AnonReq بمراحل معالجة متسلسلة:

1. **الاستلام وتوزيع المحتوى:** يدخل الطلب البوابة. يفحص موزع نوع المحتوى ترويسات الطلب ويوجه الطلب إلى المحلل المناسب (نص، JSON، أو متعدد الأجزاء).
2. **التصنيف:** يتحقق محرك التصنيف من الطلب مقابل مستويات الأمان المهيأة. إذا كان الطلب يحتوي على بيانات مقيدة غير مسموح بمرورها، فإن المحرك يمنع الطلب مبكرًا أو يحدد ما إذا كان يحتاج لتوجيه محلي أو تطهير أو مرور مباشر.
3. **الكشف:** يدمج محرك الكشف تعرفات regex/checksum وتكامل Presidio وخوارزميات تحسين السياق لتحديد الكيانات الحساسة (مثل البريد الإلكتروني وأرقام الهواتف وبطاقات الائتمان).
4. **الترميز:** يتم استخراج القيم الحساسة المكتشفة واستبدالها برموز مجهولة (مثل `[EMAIL_0]` و `[PERSON_1]`). يتم تخزين التعيينات الفريدة والعشوائية في مدير ذاكرة التخزين المؤقت Valkey/Redis مع ربطها بدورة حياة الجلسة.
5. **مهايئ موفر الخدمة:** يترجم مهايئ موفر الخدمة الطلب المطهر إلى تنسيق واجهة برمجة التطبيقات الخاص بـ LLM المستهدف (على سبيل المثال، تحويل الطلب المتوافق مع OpenAI إلى تنسيق Anthropic/Gemini) وتوجيهه.
6. **استجابة LLM:** يعيد الموديل الخارجي استجابته التي تحتوي على الرموز.
7. **الاستعادة:** يقرأ محرك الاستعادة التعيينات من مدير ذاكرة التخزين المؤقت ويستبدل الرموز بقيمها الأصلية في نص الاستجابة (يدعم استجابات البث SSE).
8. **الإخراج:** يتم إعادة الاستجابة الكاملة المستعادة إلى تطبيق العميل.

## المكونات الرئيسية

- **الوكيل/البوابة:** تطبيق FastAPI يعمل على حلقة شبكة ASGI.
- **محرك التصنيف:** نقاط القرار والإنفاذ (PDP/PEP) لتقييم سياسات الحوكمة والمخاطر.
- **محرك الكشف:** ماسح كيانات متعدد اللغات مع تعرفات regex وتكامل Microsoft Presidio.
- **محركات الترميز والاستعادة:** الأكواد المسؤولة عن تبديل واستعادة الرموز.
- **مدير ذاكرة التخزين المؤقت:** نسخة Valkey/Redis مؤقتة تحفظ التعيينات في الذاكرة تحت عقد TTL صارم.
- **مهايئات موفري الخدمة:** طبقة توافق تترجم طلبات واجهة برمجة التطبيقات في الوقت الفعلي.

## ضمانات الأمان الأساسية

- **عدم التعرض للبيانات (Zero-Exposure):** يجب ألا تعبر بيانات PII الأولية حدود شبكة المؤسسة إلى موفري الخدمة الخارجيين تحت أي ظرف من الظروف.
- **الفشل الآمن (Fail-Secure):** يجب أن يؤدي أي استثناء أثناء التشغيل أو مهلة أو عدم توفر ذاكرة مؤقتة أو غموض في التصنيف إلى حظر فوري لحركة المرور الصادرة وإرجاع استجابة HTTP 503 أو 403 إلى تطبيق العميل.
- **قياس عن بعد خالٍ من PII:** سجلات المراجعة ومقاييس Prometheus تحتوي على بيانات تعريف فقط. لا يتم تسجيل أي قيم PII أولية أو مفاتيح رموز.
- **التخزين المؤقت:** يتم تخزين التعيينات فقط في ذاكرة التخزين المؤقت مع تكوينات TTL صارمة، مما يضمن حذفها فورًا بعد انتهاء المعاملة.
