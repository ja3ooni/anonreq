    from hypothesis import given, settings, HealthCheck
    from hypothesis import strategies as st

    @settings(
        max_examples=200,
        deadline=60000,
        derandomize=True,
        suppress_health_check=[HealthCheck.too_slow, HealthCheck.data_too_large],
    )
    @given(
        email=st.emails(),

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Test framework → production code | Property tests exercise production code through the full pipeline; test harness must not mask failures |
| Failure injection → pipeline stages | Injected faults must trigger correct fail-secure paths without bypassing safety checks |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-01-01 | Tampering | Test code accepted as proof despite gap | mitigate | Systematic failure injection framework covers all 5 modes explicitly (D-162). Property-based, not example-based — Hypothesis explores the failure space. |
| T-06-01-02 | Information Disclosure | PII in test logs | mitigate | All log output captured to in-memory buffer, never written to disk. Test teardown clears buffers. |
| T-06-01-03 | Elevation of Privilege | Mock objects allowing unsafe behavior | mitigate | Mocks validated to raise proper exceptions (not return None or silently pass). Tests verify forwarded=0 via ProviderStage spy, not just absence of error. |
| T-06-01-04 | Denial of Service | ReDoS from Hypothesis-generated PII patterns | accept | PII patterns use simple character classes, not nested quantifiers. ReDoS risk is low for these synthetic inputs. |
</threat_model>
