<objective>
Implement model alias routing for the AnonReq gateway. This plan delivers: (1) the ModelAlias Pydantic model with provider, model, and future-reserved fields (capabilities, metadata, fallback, routes), (2) the AliasRegistry with YAML config loading, resolve(alias) → ModelAlias, and startup validation, (3) the GET /v1/models endpoint returning configured aliases with metadata, and (4) alias integration in the ProviderStage pipeline.

Purpose: Without model aliases, clients must specify provider-specific model names and the gateway has no abstraction layer between classification rules and upstream providers. PROV-05, PROV-07, and the routing abstraction for enterprise policy (Phase 8+) all depend on this plan.

Output: A working `GET /v1/models` that returns configured aliases, and a pipeline that resolves `model: "smart"` → `provider: "anthropic"` + `model: "claude-sonnet-4"` before calling ProviderAdapter.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client → Gateway (model field) | Client sends alias name as model field in request body |
| Config YAML → AliasRegistry | model_aliases.yaml loaded at startup defines alias→provider mapping |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-03-01 | Spoofing | Unknown alias in request | mitigate | AliasNotFoundError returns HTTP 400 with available aliases list. Never forwards request with unresolved alias. |
| T-03-03-02 | Tampering | model_aliases.yaml | mitigate | yaml.safe_load() prevents code injection. Startup validation checks all aliases reference valid providers. Hard fail on duplicate aliases. |
| T-03-03-03 | Information Disclosure | GET /v1/models response | accept | Returns alias names and metadata only. No API keys, no internal URLs, no provider credentials. Metadata is operator-configured and intended for client consumption. |
| T-03-03-04 | Tampering | Alias → Provider resolution | mitigate | AG-12: All routing through aliases. Classification rules target aliases, not provider names (D-95). This prevents bypass of routing policy. |
| T-03-03-05 | Elevation of Privilege | Future fallback routing | accept | fallback field reserved for future use. In MVP, fallback is None — no automatic failover between providers. Phase 15+ will add weighted failover with explicit governance. |
</threat_model>
