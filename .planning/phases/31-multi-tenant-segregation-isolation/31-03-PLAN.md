_known_tenants: set[str] = set()

Add `_tenant_label` function:
```python
def _tenant_label(tenant_id: str) -> str:
    """Return tenant_id or '_overflow' if cardinality limit exceeded per D-12."""
    if tenant_id in _known_tenants:
        return tenant_id
    if len(_known_tenants) >= MAX_TENANTS:
        return "_overflow"
    _known_tenants.add(tenant_id)
    return tenant_id
```

Add `set_max_tenants(max_tenants: int) -> None` function to allow Settings override at startup.

Modify the following existing metric definitions to add `"tenant_id"` as the FIRST labelname:
- `requests_total`: add `"tenant_id"` to labelnames
- `entities_detected`: no change (entity_type and locale are the relevant labels)
- `detection_latency`: no change (Histogram without labels is fine)
- `unrestored_tokens`: no change (entity_type only)
- `fail_secure_events`: add `"tenant_id"` to labelnames

- `processing_overhead`: no change (Histogram without labels is fine)
- `active_config_version`: no change (Gauge without labels is fine)
- `proxy_connections_active`: no change
- `proxy_pinning_blocks`: no change

The key insight: only the counters that are incremented per-request need tenant_id labels. Histograms and gauges that represent aggregate system state do not need tenant partitioning.
