<objective>
Build the foundation for agent and tool call governance: tool permission policy definition, multi-format tool call extraction, and PDP #2 integration for tool permission evaluation. This wave delivers: (1) Phase 8 policy YAML extension for per-provider tool governance with 4-tier permissions, (2) tool call extraction from OpenAI `tool_calls`, Anthropic `tool_use`, and MCP `tools/call` formats, (3) tool permission evaluation in PDP #2 with the 4-tier decision model, and (4) strict tool isolation between model-exposed and host-exposed tool domains with separate registries and namespaces.

Purpose: Without tool governance, the gateway has no visibility into what agents are doing — tool calls, file operations, and code execution pass through without policy enforcement. This wave makes tool calls a first-class policy decision point.

Output: Tool policies parsed from Phase 8 YAML, tool calls extracted from 3 formats, PDP #2 evaluates and returns allow/block/audit/approve decisions.
</objective>

         - `extract_calls(message: dict, format: str) -> list[ToolCall]`:
           - **OpenAI format** (`format="openai"`):
             - From `message["tool_calls"]` list in assistant message.
             - Each tool_call has `id`, `function.name`, `function.arguments` (JSON string).
           - **Anthropic format** (`format="anthropic"`):
             - From `message["content"]` list — find blocks where `type == "tool_use"`.
             - Each has `id`, `name`, `input` (dict).
           - **MCP format** (`format="mcp"`):
             - From MCP JSON-RPC `method == "tools/call"`.
             - `id` from JSON-RPC request id, `name` from `params.name`, `arguments` from `params.arguments`.
         - `extract_results(message: dict, format: str) -> list[ToolResult]`:
           - **OpenAI**: from `message["role"] == "tool"`, content in `message["content"]`.

           - **MCP**: from JSON-RPC response with `result` field containing `content`.
         - `detect_format(request_body: dict, headers: dict) -> str`:
           - Heuristic to detect which tool format is in use:
             - If `messages[].tool_calls` exists → openai.
             - If `messages[].content[].type == "tool_use"` → anthropic.
             - If `method == "tools/call"` → mcp.
         - `detect_domain(headers: dict, request_body: dict) -> str`:
           - If `X-AnonReq-Tool-Domain: host` header set → "host".
           - If MCP request from host-side MCP client → "host".
           - Otherwise → "model".

    Implement tool permission evaluation integrated into PDP #2 per D-003, D-004, D-016.

    1. **src/anonreq/governance/pdp_tool_evaluator.py:**
       - `ToolDecision` dataclass:
         - `permission: ToolPermission`, `tool_name: str`, `provider: str`.
         - `domain: str`, `risk_level: ToolRiskLevel`.
         - `reason: str | None`, `audit: dict` — details for audit log.
       - `PDPToolEvaluator` class:
         - Constructor takes `policy_parser: ToolPolicyParser`.
         - `async evaluate(tool_call: ToolCall, context: ProcessingContext) -> ToolDecision`:
           1. Detect provider and domain from context/headers.
           2. Load per-provider governance policy (from parser).
           3. Match `tool_call.name` against policy rules (exact → prefix → glob).
           4. If matched: return decision based on matched permission.
           5. If not matched:
              - If tool risk level ≥ CRITICAL and tool is unlisted → REQUIRE_HUMAN_APPROVAL.
              - Otherwise → ALLOW (explicit allowlist with default-allow for unspecified).
           6. If block: set `context.errors.append(ToolBlockedError(...))`.
           7. If require_human_approval: set `context.requires_approval = True`.
           8. Attach audit metadata to decision.
         - `evaluate_tool_result(tool_result: ToolResult, context: ProcessingContext) -> None`:
           - Validates that tool result corresponds to an approved tool call.
           - Flags orphaned tool results (no matching approved call) for audit.
         - `get_permitted_actions(provider: str, domain: str) -> list[str]`:
           - Returns list of tool names permitted for the given provider+domain.
         - **Credential isolation per D-016**:
           - Tool decisions include a `credential_context` field tracking which delegation the tool belongs to.
           - PDPToolEvaluator enforces that tool calls from one delegation never access another delegation's credentials.
           - Cross-delegation tool call → BLOCK with reason `cross_delegation_credential_leak`.
         - **Tool domain isolation per D-018, D-019, D-020**:
           - `evaluate()` rejects tool calls from model domain attempting host-domain tools.
           - `evaluate()` rejects tool calls from host domain attempting model-domain tools.
           - Isolation is structural — each domain has its own `ToolPolicyParser` instance.

    2. **config/policy.yaml — Update:**
       - Add default tool governance section:
         ```yaml
         governance:
           default_unlisted_tool_permission: allow  # allow | block
           default_critical_tool_permission: require_human_approval
           strict_domain_isolation: true
           credential_isolation: per_delegation
         ```

       - Test cross-delegation credential access → BLOCK.
