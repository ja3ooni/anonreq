<objective>
Build the async human approval flow for high-risk tool calls and the tool result inspection pipeline. This wave delivers: (1) async approval flow that suspends tool calls requiring human approval, returns HTTP 202 with an approval token, and queues the decision to the Phase 14 oversight queue, (2) approval polling endpoint at `GET /v1/oversight/approvals/{token}`, (3) tool result PII detection using the Phase 2 Detection Engine, and (4) reconstruction attempt detection that identifies attempts to regenerate PII from anonymized token patterns.

Purpose: High-risk tool calls (code execution, database queries, destructive operations) must not execute without human approval. Tool results must be inspected for sensitive data before reaching the agent. Reconstruction detection prevents the model from reversing anonymization by requesting token regeneration.

Output: Suspended tool calls routed through approval workflow; tool results inspected for PII and reconstruction attempts.
</objective>

         - `async create_approval(tool_call: ToolCall, context: ProcessingContext) -> dict`:
           - Generates: `token = secrets.token_urlsafe(32)` (256-bit entropy).
           - Creates `ApprovalRecord(status=PENDING, created_at=now, expires_at=now+ttl)`.
           - Stores in Valkey HASH keyed `anonreq:approval:{token}` with:
             - tool_call JSON, context_id, status, created_at, expires_at.

           - Saves updated record with TTL extension (60s post-decision cleanup).
           - Returns approved record + tool_call for execution.
         - `async deny_approval(token: str, decided_by: str, note: str = "") -> dict`:
           - Same flow as approve but sets status=DENIED.
         - `async cleanup_expired() -> int`:
           - Scans for expired approvals, deletes them, returns count.
         - Approval records use Valkey with the same CacheManager interface.
         - Token generation: `secrets.token_urlsafe(32)`.

    1. **src/anonreq/governance/tool_inspector.py:**
       - `InspectionResult` dataclass:
         - `tool_name: str`, `tool_id: str`.
         - `pii_detected: bool`, `pii_entity_count: int`, `pii_entity_types: list[str]`.
         - `reconstruction_attempt: bool`, `reconstruction_confidence: float`.
         - `reconstruction_indicators: list[str]`.
         - `action: str` (allow | suppress | alert).
       - `ToolResultInspector` class:
         - Constructor takes `detection_engine: DetectionEngine`, `cache_manager: CacheManager`.
         - `async inspect(tool_result: ToolResult, context: ProcessingContext) -> InspectionResult`:
           1. **PII detection**:
              - Run tool result content through Phase 2 Detection Engine.
              - If PII detected: flag `pii_detected=True`, record count and types.
           2. **Reconstruction attempt detection**:
              - Load token mappings for the session from CacheManager.
              - Scan tool result content for:
                - **Exact token pattern matches**: content containing `[TYPE_N]` patterns that match session tokens.
                - **Reconstructed values**: content containing original values that match token mappings.
                - **Regeneration prompts**: content suggesting the model was asked to reconstruct (`"regenerate the email"`, `"fill in the [NAME_0] placeholder"`).
                - **Bracket patterns**: suspicious use of bracket notation that mimics token format.
              - Score reconstruction confidence (0.0–1.0).
              - If confidence > 0.7: set `reconstruction_attempt=True`.
           3. **Action determination**:
              - If `reconstruction_attempt=True` AND confidence > 0.9 → action="suppress".
              - If `pii_detected=True` → action="alert" (suppress only if high-severity PII).
              - Otherwise → action="allow".
              - `suppress` strips the content and replaces with `[Content suppressed: tool result contained sensitive data]`.
           4. Attach `InspectionResult` to `context.audit_metadata["tool_inspection"]`.
           - Returns `InspectionResult`.

       - Test content with original value matching mapping → reconstruction_attempt=True.
       - Test content with explicit reconstruction prompt → reconstruction_attempt=True.
       - Test empty content → passes cleanly.
       - Test inspection result attaches to context.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Approval token → Client | Token exposed to client for status polling |
| Oversight queue → Human | Tool call context exposed to human approver |
| Tool result content | Raw tool result inspected — PII visible to gateway (not stored) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-18-02-01 | Spoofing | Approval token | mitigate | 256-bit random token is cryptographically unguessable. Single-use. TTL 300s. |
| T-18-02-02 | Tampering | Approval decision | mitigate | Approval records stored in Valkey with atomic operations. Only approve/deny endpoints can change status. |
| T-18-02-03 | Information Disclosure | Tool call in oversight queue | mitigate | Oversight queue stores tool name and parameters (anonymized per Phase 9). No raw entity values. |
| T-18-02-04 | Information Disclosure | Tool result PII detection | mitigate | Detection runs in-memory, results are metadata-only (entity count, types). Raw PII never logged or stored. |
| T-18-02-05 | Tampering | Reconstruction detection bypass | mitigate | Detection inspects content for 4 indicator types — bypassing one still triggers another. |
</threat_model>
