<objective>
Build the four firewall gate positions that inspect traffic at key points in the pipeline. This plan delivers: (1) Inbound pre-anon gate — scans raw input before any processing; (2) Inbound post-anon gate — scans sanitized content before ForwardingGuard; (3) Outbound pre-restore gate — scans raw provider output before token restoration; (4) Outbound post-restore gate — scans restored response before client delivery; (5) Per-severity action mapping (BLOCK/flag/forward/monitor); (6) Correct HTTP status codes for blocked requests.

Purpose: Without these gates, the rule engine is inert. The four-gate architecture ensures detection at both raw and processed layers, preventing bypass through sanitization-side effects or post-restoration reconstruction.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Raw prompt → Pre-anon gate | Untrusted user input enters first detection layer |
| Sanitized text → Post-anon gate | Processed content enters second detection layer |
| Provider output → Pre-restore gate | External LLM response enters detection layer |
| Restored response → Post-restore gate | Original values restored, enters final detection |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-10-02-01 | Tampering | Bypass pre-anon | mitigate | Post-anon gate catches injection surviving sanitization; dual-gate prevents bypass |
| T-10-02-02 | Information Disclosure | Block response | mitigate | Block responses contain reason and category only — never prompt/response text |
| T-10-02-03 | Denial of Service | Expensive outbound scan | mitigate | Configurable latency budget; truncated scan if budget exceeded |
| T-10-02-04 | Tampering | Outbound violation suppression | mitigate | Post-restore gate catches violations in restored content; dual-gate prevents bypass |
</threat_model>
