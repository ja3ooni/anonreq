| Customer internal network → Gateway | Inbound client requests carrying raw sensitive data enter the gateway trust zone |
| Gateway → External LLM provider | Sanitized-only payloads cross into untrusted external network (WAN) |
| Docker internal network (presidio, valkey) | Internal service-to-service communication within isolated bridge network |
| Log output → Docker log driver | Metadata-only structured logs leave container via stdout/stderr |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-PRR-01 | Tampering | PRR.md, DEPLOYMENT_GUIDE.md | mitigate | All env vars and config values verified against architecture guardrails and requirements; cross-reference table links each doc to source-of-truth |
| T-PRR-02 | Information Disclosure | THREAT_MODEL.md | mitigate | Threat model explicitly documents which data crosses each trust boundary; mitigations reference specific code mechanisms (not generic advice) |
| T-PRR-03 | Elevation of Privilege | DEPLOYMENT_GUIDE.md | mitigate | Deployment guide specifies internal network isolation, no exposed presidio/valkey ports, Valkey requirepass |
| T-PRR-04 | Denial of Service | RUNBOOK.md, SRE_PLAYBOOK.md | mitigate | Runbook covers resource sizing, scaling limits, backpressure; SRE playbook has incident procedures for degradation |
