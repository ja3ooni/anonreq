<objective>
Wire the Enterprise Policy Engine into the gateway request pipeline. This plan delivers: (1) PolicyDecisionPoint — evaluates classification, rate limit, spend, and residency rules against request context; (2) PolicyEnforcementPoint — translates decisions into HTTP 402/429/451/503 responses or forwarding allowances; (3) ForwardingGuard — final validation that no request bypasses policy; (4) decision caching for hot-path efficiency; and (5) fail-closed integration — any policy evaluation failure returns 503 with zero provider forwarding.

Purpose: Without gateway orchestration, the policy models from 08-01 are inert data structures. This plan makes them active enforcement points in the request lifecycle.

Output: All requests passing through PDP → PEP → ForwardingGuard with deterministic policy enforcement before provider contact.
</objective>

       - `PolicyDecisionPoint` class:
         - Constructor: `__init__(self, policy_store: PolicyStore, usage_limiter: UsageLimiter, spend_controller: SpendController, residency_router: ResidencyRouter)`
         - `async def evaluate_all(self, ctx: ProcessingContext) -> PolicyDecision`:
           - Run checks in order: classification → rate_limit → spend → residency
           - Fail-fast: first BLOCK decision short-circuits remaining checks
           - Cache decision in memory dict with TTL (default 5s) keyed by tenant_id + request_hash
           - Return ALLOW if all checks pass
         - `async def evaluate_classification(self, tenant_id: str, classification: str) -> PolicyDecision`:
           - Load applicable rules from PolicyStore
           - Find highest-priority matching rule based on classification level
           - Return rule action (BLOCK, ALLOW, FLAG_AND_FORWARD)
         - `async def evaluate_rate_limit(self, tenant_id: str) -> PolicyDecision`:
           - Delegate to UsageLimiter.check_rate_limit()
         - `async def evaluate_spend(self, tenant_id: str) -> PolicyDecision`:
           - Delegate to SpendController.check_spend()
         - `async def evaluate_residency(self, tenant_id: str, provider: str, region: str) -> PolicyDecision`:
           - Delegate to ResidencyRouter.resolve_region()
         - `def _compute_request_hash(self, ctx: ProcessingContext) -> str`:
           - SHA-256 of sanitized request components (model, tenant_id, classification)
           - Used for decision cache key

    2. **tests/policy/test_pdp.py**:

       - Test each evaluation method in isolation

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| HTTP Request → PDP | Untrusted request data enters policy evaluation |
| PDP decision cache → PEP | Cached decision drives enforcement |
| ForwardingGuard → Provider | Guard output determines whether provider receives request |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-08-02-01 | Elevation of Privilege | PDP evaluates all requests | mitigate | PDP runs before any route handler; classification rules gate all requests |
| T-08-02-02 | Tampering | Decision cache | mitigate | TTL-bounded cache (5s default); expired cache forces re-evaluation |
| T-08-02-03 | Tampering | ForwardingGuard bypass | mitigate | Guard is final check in provider routing function; no code path skips it |
| T-08-02-04 | Denial of Service | Expensive evaluation loops | mitigate | Fail-fast on first BLOCK; decision caching; bounded rule evaluation per request |
| T-08-02-05 | Information Disclosure | Transparency headers | mitigate | Headers contain counts and reason codes only; no raw entity values, tokens, or prompts |
</threat_model>
