    The test generates N independent sessions (N ≥ 1000), processes text containing the same entity value through each session, collects all resulting tokens, and asserts they are all unique.

    ```python
    import uuid
    from hypothesis import given, settings, HealthCheck, assume
    from hypothesis import strategies as st
    from tests.property.strategies import pii_text_strategy

    MAX_SESSIONS = 1000

    @settings(
        max_examples=200,
        deadline=120000,      # 120s — generating 1000 sessions takes time
        derandomize=True,
        suppress_health_check=[HealthCheck.too_slow, HealthCheck.data_too_large],
    )
    @given(
        entity_value=st.emails(),
        num_sessions=st.integers(min_value=MAX_SESSIONS, max_value=MAX_SESSIONS + 100),
    )
    async def test_cross_request_token_randomization(entity_value, num_sessions):
        """Same entity value across independent sessions produces different tokens."""
        text = f"My contact email is {entity_value}"
        all_tokens = set()

        for i in range(num_sessions):
            # Create a new processing context with a unique session seed
            session_id = str(uuid.uuid4())
            context = ProcessingContext(
                request_id=f"req_{i}",
                session_id=session_id,
                # The TokenizationStage uses a per-session random seed derived from
                # the session_id to generate token offsets per TOKN-05
            )

            # Run the pipeline for this session
            result = await process_request(
                test_client=test_client,
                payload={
                    "model": "gpt-4",
                    "messages": [{"role": "user", "content": text}],
                },
                context=context,
            )

            # The entity value should be detected and tokenized
            # Get the token for our entity value from token_mappings
            # token_mappings is dict[original_value, token]
            entity_token = None
            for orig_val, token in result.token_mappings.items():
                if entity_value in orig_val:
                    entity_token = token
                    break

            assume(entity_token is not None)  # Skip if entity not detected

            # Verify: this token has not been seen in any previous session
            assert entity_token not in all_tokens, (
                f"Token collision: {entity_token} for value {entity_value} "
                f"appeared in session {all_tokens[entity_token]} and session {i}"
            )
            all_tokens.add(entity_token)

        # Verify collision bound: with N=1000 and 2^32 possible offsets,
        # P(collision) ≈ N² / (2 × 2³²) ≈ 0.00012
        # Expected collisions at this bound: ~0. Over 200 Hypothesis examples,
        # expected total collisions across all runs: ~0.024
        # We assert exactly 0 collisions.
        assert len(all_tokens) == num_sessions, (
            f"Expected {num_sessions} unique tokens, got {len(all_tokens)}"
        )
    ```

    The test depends on the TokenizationStage using a per-session random seed per TOKN-05. The seeding mechanism (D-176) uses:
    - Session seed: UUIDv7 (128-bit, temporally ordered)
    - Token index derivation: `SHAKE_128(seed + entity_value + entity_type, 4 bytes)` → 32-bit offset
    - Per-type counter maps offset to `[TYPE_{N}]` where N starts at a random position

    If the seeding mechanism is not yet implemented in the TokenizationStage (it was specified in Phase 2 but may have been deferred), this test surfaces that gap. The test does not mock the seeding — it tests the real production implementation.

    Test organization:
    - One test function with Hypothesis @given for different entity types
    - Entity types: email, phone, credit card, name, IBAN
    - Each entity type gets its own test function with appropriate strategy
    - Each test generates `num_sessions` independent calls through the full pipeline
    - Each session has a unique session_id (simulating independent requests)
    - Tokens collected across sessions must all be unique

    Hypothesis settings per TEST-PLAN.md:
    - max_examples=200
    - deadline=120000 (2 min per example — 1000 sequential pipeline invocations)
    - derandomize=True
    - suppress_health_check as needed

    Performance note: 1000 sequential pipeline invocations is slow. Consider using smaller N (e.g. 100) for quick Hypothesis runs and a separate pytest.mark.slow test for the full 1000-session proof. Document this in the test file header.
