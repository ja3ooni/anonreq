    1. **src/anonreq/models/dsar.py**:
       - `DsarRequestType` enum: `ERASURE`, `RESTRICTION`, `RECTIFICATION`, `PORTABILITY`, `ACCESS`
       - `SubjectStatus` enum: `ACTIVE`, `DELETED`, `PROCESSING_RESTRICTED`, `LEGAL_HOLD`
       - `DsarRequest(BaseModel)`: id (str), tenant_id (str), subject_id (str), request_type (DsarRequestType), status (str="pending_verification"), verification_details (dict | None), result (SubjectStatus | None), submitted_at (datetime), verified_at (datetime | None), fulfilled_at (datetime | None), notes (str | None)
       - `DsarResult(BaseModel)`: request_id (str), subject_status (SubjectStatus), summary (str), fulfilled_at (datetime)

    2. **src/anonreq/dsar/workflow.py**:
       - `class DsarWorkflow`:
         - `__init__(self, db: AsyncSession, cache_manager, legal_hold_manager, restriction_service)`
         - `async def submit_request(self, tenant_id: str, subject_id: str, request_type: DsarRequestType, verification_details: dict | None = None) -> DsarRequest`:
           - Creates request with status="pending_verification"
           - Emits `dsar_request_received` audit event
         - `async def verify_request(self, request_id: str, verified_by: str) -> DsarRequest`:
           - Sets status="processing", verified_at, verified_by
         - `async def fulfill_request(self, request_id: str, fulfilled_by: str) -> DsarResult`:
           - Loads request, checks Legal Hold via `legal_hold_manager.is_on_hold(tenant_id, record_id=subject_id)`
           - If hold active: returns SubjectStatus.LEGAL_HOLD
           - If request_type == ERASURE:
             - Calls `erase_subject_data(subject_id)` to delete Valkey mappings
             - Sets subject_status = DELETED
           - If request_type == RESTRICTION:
             - Calls `restrict_subject(tenant_id, subject_id)` to block future requests
             - Sets subject_status = PROCESSING_RESTRICTED
           - Sets status="fulfilled", fulfilled_at
           - Emits `dsar_completed` audit event
           - Returns DsarResult
         - `async def get_request_status(self, request_id: str) -> DsarRequest`

    3. **Add to governance router**:
       - `POST /v1/admin/dsar` — submit DSAR request
       - `GET /v1/admin/dsar` — list DSAR requests
       - `GET /v1/admin/dsar/{id}` — get request status
       - `POST /v1/admin/dsar/{id}/verify` — verify request
       - `POST /v1/admin/dsar/{id}/fulfill` — fulfill request

    4. **tests/test_dsar.py**:

  <action>
    Implement erasure and restriction services per D-022, D-023.

    1. **src/anonreq/dsar/erasure.py**:
       - `class DataErasureService`:
         - `__init__(self, cache_manager)`: uses CacheManager to access Valkey
         - `async def erase_subject_data(self, subject_id: str) -> bool`:
           - Scans Valkey for keys matching patterns containing subject_id reference
           - Deletes all token→entity mappings for this subject
           - Stores erasure record in PostgreSQL: `{"subject_id": str, "erased_at": datetime}`
           - Idempotent: if already erased, returns True without error
           - Returns True if data found and erased, False if no data to erase
         - `async def has_been_erased(self, subject_id: str) -> bool`:
           - Checks PostgreSQL erasure record

    2. **src/anonreq/dsar/restriction.py**:
       - `class DataRestrictionService`:
         - `__init__(self, db: AsyncSession, cache_manager)`
         - `async def restrict_subject(self, tenant_id: str, subject_id: str) -> bool`:
           - Stores restriction record in PostgreSQL: `{"tenant_id", "subject_id", "restricted_at"}`
           - Also caches in Valkey (fast-path check for pipeline): `anonreq:restricted:{subject_id}`
           - Returns True if newly restricted, False if already restricted
         - `async def is_subject_restricted(self, subject_id: str) -> bool`:
           - Checks Valkey cache first (fast path), falls back to PostgreSQL
           - Returns True if subject is restricted
         - `async def remove_restriction(self, subject_id: str) -> bool`:
           - Deletes restriction record from PostgreSQL and Valkey cache
           - Returns True if restriction was active

    3. **Wire restriction into pipeline** (entry check):
       - In pipeline orchestration (Phase 2, before classification):
         - Check `is_subject_restricted(subject_id)` for identifiable subjects
         - If restricted: return HTTP 451 (Unavailable For Legal Reasons) with `{"error": "subject_processing_restricted"}`

    4. **tests** (extends test_dsar.py):
       - Test erasure of Valkey mappings
       - Test idempotent erasure
       - Test restriction check fast-path (Valkey)
       - Test restriction blocks pipeline entry
       - Test restriction removal
  </action>
