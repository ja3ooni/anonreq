<objective>
Build the transparent proxy foundation for the Universal AI Traffic Gateway — MITM TLS termination with tenant-managed CA certificates. This wave delivers: (1) TLS interception layer that terminates client TLS using tenant CA and re-originates to upstream providers, (2) dual-path CA certificate management via admin API and filesystem file watch with hot-reload, (3) certificate pinning detection that identifies clients using pinned certificates and blocks forwarding, and (4) MITM FastAPI middleware that integrates with the existing gateway pipeline.

Purpose: Without TLS interception, the gateway can only inspect traffic that is explicitly routed through it as a reverse proxy. Transparent proxy mode requires TLS termination so the gateway can inspect encrypted traffic between client applications and AI providers. The dual-path CA management ensures operators can provision certificates via automation (API) or manual ops (file drop).

Output: Gateway running in transparent proxy mode terminates TLS using tenant CA, re-originates to providers, and blocks pinned-certificate clients with HTTP 426.
</objective>

         - Loads CA cert and key from PEM files. If no CA configured, raises `ConfigurationError`.
         - Creates an `ssl.SSLContext` for client-facing TLS termination using `ssl.PROTOCOL_TLS_SERVER`.

         - Factory function that returns a configured server-side TLS context.
         - Sets `check_hostname=False` on client-facing context (we terminate, not validate client).
         - Sets secure cipher suites: `TLS_AES_256_GCM_SHA384:TLS_CHACHA20_POLY1305_SHA256:ECDHE-ECDSA-AES128-GCM-SHA256`.

         - Returns client-side TLS context for re-origination to providers.
         - Sets `check_hostname=True`, loads default system CAs.

         - Static method that checks if a client certificate contains known pinning indicators:

           - Certificate validity period matches known pinned cert profiles.
         - Returns True if pinning indicators detected — MITM handler will block.
       - `reload_ca(ca_cert_path: str, ca_key_path: str) -> None`:
         - Replaces CA cert/key and recreates SSL context. Called by hot-reload.
       - Use Python stdlib `ssl`, `cryptography` for cert parsing.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client → Gateway TLS | Untrusted client TLS connection terminated at gateway |
| Gateway → Provider TLS | Gateway re-originates TLS to upstream provider |
| CA cert storage | CA cert/key PEM files on filesystem |
| API → CAManager | CA cert upload via admin API crosses network boundary |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-17-01-01 | Spoofing | TLS termination | mitigate | Gateway validates client cert against tenant CA before re-origination. Pinning detection blocks pinned clients. |
| T-17-01-02 | Tampering | CA cert files | mitigate | Filesystem permissions on CA directory (owned by anonreq:anonreq, 0600). File watch detects unauthorized changes. |
| T-17-01-03 | Information Disclosure | TLS re-origination | mitigate | Upstream context validates provider certs via system CA bundle. No downgrade to insecure ciphers. |
| T-17-01-04 | Denial of Service | MITM tunnel pool | mitigate | Configurable max concurrent tunnels. Idle timeout (default 300s). Tunnel count exposed via Prometheus `anonreq_proxy_connections_active`. |
| T-17-01-05 | Elevation of Privilege | CA upload API | mitigate | Admin-only auth (Phase 1 static bearer token). Validates cert+key pair match before accepting. Limits cert key size to 4096-bit. |
</threat_model>
