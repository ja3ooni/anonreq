<objective>
Deliver the remaining Phase 17 capabilities: proxy-only mode for low-overhead routing, performance optimization to meet P95 < 5ms / P99 < 10ms targets, and VM appliance packaging for enterprise deployment. This wave delivers: (1) proxy-only mode that skips all detection/anonymization and provides minimal-latency routing + TLS + audit, (2) performance profiling and optimization to meet latency targets, (3) VM image packaging via Packer with the same Docker Compose services plus systemd, and (4) appliance management agent for local health, config, and update operations.

Purpose: Without proxy-only mode, operators pay full detection/anonymization latency for every request even when only routing is needed. Without VM packaging, enterprise operators must manually provision Docker hosts. The appliance management agent provides the Day-2 operations interface.

Output: Gateway runs in proxy-only mode at P95 < 5ms, ships as a bootable VM image with management agent.
</objective>

         ```python
         class ProxyMode(str, enum.Enum):
             PROXY_ONLY = "proxy-only"
             FULL = "full"
             TRANSPARENT = "transparent"
         ```
       - `get_pipeline_for_mode(mode: ProxyMode) -> list[str]`:
         - Returns ordered list of pipeline stage names for the given mode:
           - `PROXY_ONLY`: `["auth", "routing", "forwarding_guard", "audit"]`.
           - `FULL`: `["auth", "routing", "classification", "detection", "anonymization", "forwarding_guard", "provider_call", "restoration", "audit"]`.
           - `TRANSPARENT`: same as FULL (MITM is a transport, not a pipeline mode).
       - `requires_mitm(mode: ProxyMode) -> bool`:
           - Returns True for TRANSPARENT.
       - `requires_detection(mode: ProxyMode) -> bool`:
           - Returns True for FULL and TRANSPARENT, False for PROXY_ONLY.
       - `mode_from_env() -> ProxyMode`:
           - Reads `ANONREQ_PROXY_MODE` from environment, validates, returns enum.
           - Raises `ConfigurationError` on invalid value.
       - `proxy_mode_description(mode: ProxyMode) -> str`:
           - Returns human-readable description of each mode.

       - In `create_app()`:
         - Read `settings.PROXY_MODE` (new field, defaults to `"full"`).
         - At startup, log the active mode: `"AnonReq starting in {mode} mode"`.
         - Based on mode:
           - `PROXY_ONLY`: register auth middleware + MITM handler + ForwardingGuard. Skip Detection Engine, Classification Engine, Tokenization Engine, Restoration Engine.
           - `FULL`: register all pipeline stages as before.
           - `TRANSPARENT`: same as FULL + MITM CONNECT handling.
         - Mode is immutable after startup — no runtime mode switch.

         - Exposes `.elapsed_ms` property.
       - `register_latency_metric(stage: str, elapsed_ms: float) -> None`:
         - Updates Prometheus histogram `anonreq_proxy_latency_ms` with stage and mode labels.
       - `_optimize_middleware_chain()`:
         - In proxy-only mode, minimize middleware count:
           - Skip CORS, skip compression, skip content-type inspection.
           - Only: `AuthMiddleware`, `MITMHandler`, `ForwardingGuard`.
           - Each middleware is instrumented with `LatencyTimer`.
       - `ConnectionPoolConfig`:
         - Configure upstream connection pooling: keepalive, max_connections=100, idle_timeout=30s.
         - HTTP/2 support for provider connections (if `httpx` supports).
       - `RequestFastPath`:
         - For proxy-only: bypass request body parsing entirely — stream raw bytes.
         - For full: parse body once, share with all pipeline stages.
         - Use `request.stream()` instead of `request.json()` when body not needed.

    2. **Performance profiling approach:**
       - Add `--profile` flag or `ANONREQ_PROFILE=1` env var that enables:
         - Per-request stage timing logged at DEBUG level.
         - `anonreq_proxy_latency_ms` histogram with P50/P95/P99 labels.
       - Instrumentation at these checkpoints:
         - `request_received`: ASGI receive event.
         - `auth_complete`: after auth middleware.
         - `proxy_decision`: after ForwardingGuard.
         - `upstream_connect`: after upstream TLS established.
         - `response_sent`: after first byte to client.
       - Metrics exposed via existing `/metrics` endpoint (Phase 5).

           - Loads appliance config from `config_path`.
         - `async get_health() -> dict`:
           - Reports: Docker Compose service statuses, disk usage, memory, uptime.

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-17-03-01 | Tampering | Mode selection | mitigate | Mode read from env var at startup only, immutable at runtime. Proxy-only mode still has full audit trail — cannot bypass routing+auth. |
| T-17-03-02 | Elevation of Privilege | Appliance agent | mitigate | Agent runs as `anonreq` non-root user. Only docker compose subcommands allowed — no arbitrary shell. All commands logged. |
| T-17-03-03 | Denial of Service | Proxy-only performance | mitigate | LatencyTimer uses `perf_counter_ns` — overhead < 100ns. Middleware chain minimized for proxy-only mode. Connection pooling prevents connection exhaustion. |
| T-17-03-04 | Tampering | VM image build | mitigate | Packer script validates checksums of all downloaded packages. Docker images built from source before packaging. Script checks GPG signatures of Docker packages. |
</threat_model>
