<objective>
Implement the Tokenization Engine — the component that replaces detected PII spans with `[TYPE_N]` placeholders. Delivers: token generation with deduplication (same value → same token), per-type atomic counters, reverse-offset replacement to prevent position drift, cryptographically random seed offsets per session, and the no-entities fast path (forward unchanged, no mapping).

Purpose: Tokenization is the core anonymization mechanism. The tokens must be deterministic within a session (deduped), non-colliding across types (different entity types have independent counters), and position-safe (reverse-offset prevents garbled output).

Output: A working `Tokenizer` where `tokenize(text, detections)` returns `(tokenized_text, mapping_dict)` satisfying all TOKN invariants.
</objective>

    1. **src/anonreq/tokenization/__init__.py**:

    2. **src/anonreq/tokenization/tokenizer.py** — per RESEARCH Pattern 4:
       - Module-level `TOKEN_PATTERN = re.compile(r'\[([A-Z][A-Z_]{0,19})_(\d+)\]')` per TOKN-01
       - `class Tokenizer`:
         - Constructor initializes per-session state:
           - `self._per_type_counters: dict[str, int] = {}` — atomic counter per entity type
           - `self._value_to_token: dict[str, str] = {}` — dedup map (value → token)
           - `self._seed: int = 0` — set during initialize_session()
         - `initialize_session() -> None`:
           - Reset all counters and maps
           - Generate new seed: `self._seed = secrets.randbits(32)` per TOKN-05
           - `secrets.randbits(32)` from stdlib `secrets` module provides cryptographic randomness
         - `tokenize(text: str, detections: list[dict]) -> tuple[str, dict[str, str]]`:
           - If not detections: return (text, {}) per TOKN-06/07
           - Sort detections descending by start position: `sorted(detections, key=lambda s: s["start"], reverse=True)` per TOKN-04
           - Initialize tokenized = text, mapping = {}
           - For each detection (sorted descending):
             - Extract original_value = text[span["start"]:span["end"]]
             - Check dedup: if original_value in self._value_to_token → reuse token
             - Else:
               - Truncate entity_type to 20 chars: `entity_type_short = span["entity_type"][:20]` per TOKN-01
               - Get counter for this type (default 0)
               - Compute token_index = (self._seed & 0x3FFFFFFF) + counter per TOKN-05
               - Generate token = f"[{entity_type_short}_{token_index}]"
               - Store in dedup map: `self._value_to_token[original_value] = token`
               - Increment per-type counter: `self._per_type_counters[entity_type_short] = counter + 1`
               - Store in mapping: `mapping[token] = original_value`
             - Reverse-offset replacement (right-to-left since sorted descending):
               - `tokenized = tokenized[:span["start"]] + token + tokenized[span["end"]:]`
           - Return (tokenized, mapping)
         - `get_mapping() -> dict[str, str]`:
           - Return current session's value_to_token mapping (for Valkey storage)
           - Return `self._value_to_token.copy()` to prevent external mutation

       The reverse-offset approach is critical per RESEARCH Pitfall 1: "Replace spans from left to right changes character offsets." Sorting descending by start position ensures earlier replacements (left side of text) are applied last, so their position hasn't shifted.

       The entity type string from Presidio may be like "PERSON" or "EMAIL_ADDRESS" — these are under 20 chars. Only custom enterprise patterns could exceed 20 chars, and the truncation ensures TOKN-01 compliance.

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-03-01 | Information Disclosure | Token seed prediction | mitigate | `secrets.randbits(32)` uses cryptographically secure RNG per TOKN-05. Different seed per session ensures cross-session token indices differ. ASVS V6 Cryptography: CSPRNG for security-sensitive random values. |
| T-02-03-02 | Tampering | Reverse-offset position drift | mitigate | Sort spans descending by start position before replacement per TOKN-04 and RESEARCH Pitfall 1. This prevents earlier replacements from shifting later span coordinates. |
| T-02-03-03 | Tampering | Token format collision | mitigate | Per-type independent counters prevent cross-type collisions (EMAIL_0 vs PHONE_0). Seed + counter ensures intra-type uniqueness. Entity type truncated to 20 chars max per TOKN-01. |
| T-02-03-04 | Information Disclosure | Token type reveals entity category | accept | Token type (EMAIL_ADDRESS, PERSON, etc.) reveals the category of detected PII. This is inherent to the `[TYPE_N]` format and is an intentional design choice per requirements. Tokenized requests are sent to external LLM providers — the entity type information is needed for debugging and transparency. |
| T-02-03-05 | Denial of Service | Infinite loop via corrupted detections | mitigate | Tokenizer iterates over a fixed list of detections. Each iteration does O(1) string slicing. No recursion or unbounded loops. Guard: if start > end or start < 0, skip the detection. |
| T-02-03-SC | Tampering | No new packages in this plan | accept | Plan 02-03 only uses stdlib modules (re, secrets, typing). No new external dependencies. |
</threat_model>
