<objective>
Implement the compliance preset engine for the AnonReq gateway. This plan delivers: (1) CompliancePreset Pydantic model with mandatory entity types, confidence thresholds, and minimum recognizer tiers, (2) 6 compliance preset YAML files (GDPR, LGPD, PDPA, POPIA, Privacy Act AU, PIPEDA), (3) PresetEngine that loads presets from YAML, merges them (Base -> Preset -> Overrides), and validates, (4) startup validation that hard-fails if active presets' mandatory requirements are not met -- exit code 1, (5) merge logic: union entity types, highest threshold, never weaken (AG-14), (6) compliance_preset field in audit log, (7) GET /v1/compliance/presets endpoint, and (8) health check inclusion.

Purpose: Without compliance presets, the gateway cannot enforce jurisdiction-specific PII detection requirements. COMP-01 through COMP-05 and audit logging compliance fields all depend on this plan.

Output: A working PresetEngine that merges 6 presets at startup, validates the effective config, hard-fails on violations, and serves GET /v1/compliance/presets.
</objective>

    Implement PresetEngine, merge logic, and startup validation per D-124 through D-132 and ARCHITECTURE.md.

    1. **src/anonreq/compliance/merge.py**:
       - `@dataclass class PresetMergeResult`:
         - merged_entity_types: dict[str, RecognizerTier]
         - merged_thresholds: dict[str, float]
         - merged_minimum_tiers: dict[str, list[RecognizerTier]]
         - requires_checksum: list[str]
         - source_presets: list[str]
         - has_customer_overrides: bool = False
       - `def merge_presets(base_config: dict, presets: list[CompliancePreset], overrides: dict | None = None) -> PresetMergeResult`:
         - Start with base_config entity types and thresholds
         - For each preset (in order):
           - Union: add any new entity types not already in merged set
           - For existing entity types: keep higher threshold (never lower)
           - For minimum tiers: union of tier requirements (if one preset needs REGEX+NER and another needs just REGEX, the result is REGEX+NER)
           - Union requires_checksum lists
         - Apply customer overrides on top (same rules: never weaken)
         - Return PresetMergeResult with source tracking

    2. **src/anonreq/compliance/validation.py**:
       - `@dataclass class ComplianceViolation`:
         - preset_id: str
         - entity_type: str
         - violation_type: str  # "missing_type", "low_threshold", "missing_tier"
         - message: str
       - `def validate_effective_config(active_presets: list[CompliancePreset], effective_config: dict) -> list[ComplianceViolation]`:
         - For each active preset:
           - For each mandatory_entity_type:
             - Check entity type exists in effective_config entity types
             - Check confidence threshold >= preset threshold
             - Check required recognizer tiers are present
             - If entity in preset.requires_checksum: check checksum validator registered
           - Collect all violations (do not fail-fast)
         - Return list of violations (empty = pass)

         - `def list_presets(self) -> dict[str, CompliancePreset]`
         - `def merge(self, active_preset_ids: list[str], base_config: dict, overrides: dict | None = None) -> PresetMergeResult`:

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-03-01 | Tampering | Preset YAML loading | mitigate | yaml.safe_load() prevents code injection. CompliancePreset model validation ensures correct types. Invalid presets cause startup validation failure. |
| T-04-03-02 | Elevation of Privilege | AG-14 invariant violation | mitigate | Merge logic enforces never-weaken: entity types never removed (only added), thresholds never lowered (only raised), tiers never downgraded (only upgraded). Unit tests verify invariants. Property tests verify associativity. |
| T-04-03-03 | Tampering | Startup validation bypass | mitigate | D-129: hard fail -- gateway won't serve traffic if active preset's mandatory types are disabled. assert_startup_checks calls sys.exit(1) on any violation. Cannot be bypassed without code change. |
| T-04-03-04 | Information Disclosure | compliance_preset audit field | mitigate | D-137: only preset names logged (comma-separated). Preset internals (thresholds, entity lists) not logged. Metadata-limited per AUDT-02. |
| T-04-03-05 | Denial of Service | Too many active presets | accept | Active preset list is configured by operator, not client-controlled. Merge of 20+ presets is O(n*m) but n and m are small (< 20 types, < 10 presets). Negligible startup cost. |
| T-04-03-06 | Tampering | Customer overrides weaken preset | mitigate | D-125: customer overrides applied after presets, but merge logic enforces never-weaken for thresholds and tiers. Overrides can add types but not remove preset-mandated types. |
</threat_model>
