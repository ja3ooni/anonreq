<objective>
Implement the multi-provider adapter layer for the AnonReq gateway. This plan delivers: (1) the ProviderAdapter abstract base class with ProviderRequest/Response/Result types and ProviderCapabilities model, (2) ProviderRegistry for YAML-based adapter resolution, (3) CapabilityResolver for startup-cached capability lookup, (4) AnthropicAdapter with OpenAI→Anthropic message translation, execute, stream_events normalization, and error handling, (5) GeminiAdapter with OpenAI→Gemini contents[] translation, and (6) OllamaAdapter with thin passthrough for OpenAI-compatible schema. All adapters normalize provider errors per PROV-08.

Purpose: Without provider adapters, the gateway can only communicate with OpenAI-API-compatible providers. All multi-provider support (PROV-02 through PROV-08) and the streaming pipeline (03-01) depend on this layer.

Output: A working `ProviderRegistry.get_adapter("anthropic")` returning an AnthropicAdapter that translates OpenAI requests to Anthropic format, executes them, and normalizes responses/streams/errors back to canonical format.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Gateway ↔ Provider API | Provider request/response crosses network boundary to external LLM API |
| Config YAML → ProviderRegistry | YAML config resolved at startup for adapter mapping |
| Env vars → API key resolution | Provider API keys from environment at startup |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-02-01 | Information Disclosure | API key in ProcessingContext | mitigate | AG-09: Secrets never enter ProcessingContext. API keys resolved only inside ProviderFactory/ProviderAdapter at network boundary. ProviderRequest headers include key, but key never logged or stored. |
| T-03-02-02 | Information Disclosure | Provider error messages | mitigate | PROV-08: Provider errors forwarded with generic messages. All raw error bodies stripped of keys, URLs, and content before returning to client. ADAPT-04 tests verify this invariant. |
| T-03-02-03 | Tampering | ProviderAdapter schema translation | mitigate | translate_request is pure function — no side effects, no state mutation. ProcessingContext unchanged by translation. AG-01: no policy, detection, or cache logic in adapters. |
| T-03-02-04 | Spoofing | Unregistered provider | mitigate | ProviderRegistry raises ProviderNotFoundError for unknown providers. Prevents routing to unconfigured endpoints. |
| T-03-02-05 | Tampering | ProviderRegistry YAML loading | mitigate | yaml.safe_load() prevents arbitrary code execution. Config validated at startup — hard fail if adapter class not importable. |
| T-03-02-06 | Information Disclosure | StreamEvent metadata | accept | StreamEvent.metadata dict may contain provider-specific diagnostic info. In MVP, metadata keys are allowlisted. Future: enforce metadata field allowlist. |
| T-03-02-07 | Elevation of Privilege | Adapter capabilities | mitigate | CapabilityResolver reads from YAML (authoritative). Future provider discovery (D-74) is validate-only — never overrides YAML config. |
| T-03-02-SC | Tampering | Provider HTTP clients | mitigate | All adapter HTTP calls use httpx with timeout (default 30s). Connection errors caught and normalized. No raw provider connection info in error logs. |
</threat_model>
