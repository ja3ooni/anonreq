<objective>
Build the Secure RAG ingestion pipeline. This wave delivers: (1) `POST /v1/rag/ingest` endpoint that accepts documents for anonymized ingestion, (2) document chunker with configurable boundaries (paragraph, sentence, token) and entity-boundary awareness, (3) ingestion-time detection running the Phase 2 Detection Engine on each chunk, (4) ingestion-time anonymization tokenizing detected entities before storage, (5) chunk metadata schema storing classification_level, entity_types_present, source_app_id, original_doc_id, (6) vector store connector interface with pluggable backends (Pinecone, Weaviate, Chroma, pgvector), and (7) RAG ingestion audit events and Prometheus metrics.

Purpose: Without Secure RAG ingestion, sensitive data flows into vector stores unanonymized — a permanent data leak since vector embeddings cannot be easily purged.

Output: Documents ingested through `/v1/rag/ingest` are chunked, inspected for PII, anonymized, and stored in the configured vector store with classification metadata.
</objective>

    2. **src/anonreq/rag/chunker.py:**
       - `ChunkBoundary` enum: `PARAGRAPH`, `SENTENCE`, `TOKEN_COUNT`.
       - `Chunk` dataclass:
         - `index: int`, `text: str`, `start_char: int`, `end_char: int`.
         - `tokens: int` (approximate token count).
       - `DocumentChunker` class:
         - Constructor takes `boundary: ChunkBoundary = ChunkBoundary.PARAGRAPH`.
           - `max_chunk_size: int = 512` (in tokens), `overlap: int = 0`.
           - `entity_aware: bool = True`.
         - `chunk(text: str, entity_spans: list[tuple[int, int]] | None = None) -> list[Chunk]`:
           - **Paragraph mode**: split on `\n\n` occurrences.
           - **Sentence mode**: split on sentence boundaries using regex `(?<=[.!?])\s+`.
           - **Token mode**: count words as approximate tokens, split at max_chunk_size boundary.
           - **Entity awareness** (`entity_aware=True`):
             - Accept `entity_spans` as `[(start, end), ...]` — from pre-scan or caller.
             - When a split point falls inside an entity span, adjust split to before entity start.
             - Ensures that `[EMAIL_0]` or any entity is not split across two chunks.
           - Reassemble: chunks are contiguous and cover the full document.
           - No character is dropped — all chunks combined = full document.
         - `estimate_tokens(text: str) -> int`:
           - Simple word-count based estimate (len(text.split())).

    3. **tests/test_rag_chunker.py:**
       - Test paragraph split: `"Para1\n\nPara2\n\nPara3"` → 3 chunks.
       - Test sentence split: `"First. Second! Third?"` → 3 chunks.
       - Test token-count: 2000 word doc with max=512 → 4 chunks.
       - Test entity awareness: entity at chars 50-60 → no split at 55.
       - Test overlap: overlap=20 chars produces overlapping chunks.
       - Test empty text → empty list.

    The chunker is synchronous and stateless. Entity spans from pre-scan (or caller).

       - `IngestRequest(BaseModel)`:

         - `metadata: dict` — source_app_id, original_doc_id, business_unit, author (optional).
         - `chunk_config: dict | None` — boundary, max_chunk_size, overlap overrides.
       - `IngestResponse(BaseModel)`:
         - `chunks: list[dict]` — each chunk with id, text (anonymized), metadata.

       - `RAGIngestionService` class:
         - Constructor takes `detection_engine`, `tokenization_engine`, `chunker: DocumentChunker`, `vector_connector: VectorStoreConnector`, `audit_logger`.
         - `async ingest(request: IngestRequest, context: ProcessingContext) -> IngestResponse`:
           1. **Pre-scan**: Run Detection Engine on full document to get entity spans.
           2. **Chunk**: Use chunker with entity spans to create entity-aware chunks.
           3. **Detect**: Run Detection Engine on each chunk text.
           4. **Classify**: Run Classification Engine on each chunk (Phase 12).
           5. **Anonymize**: Run Tokenization Engine on each chunk to replace entities with `[TYPE_N]`.
           6. **Metadata**: Generate `ChunkMetadata` per chunk.
           7. **Store**: `vector_connector.embed_and_store(anonymized_chunks, metadata)`.
           8. **Audit**: Emit `rag_content_anonymized` event.
           9. **Metrics**: Increment counters.
           10. Return response with anonymized chunks and metadata.
         - Note: Token mappings are session-scoped — stored in CacheManager with session TTL.
         - Each chunk gets a unique `chunk_id = f"{original_doc_id}:{chunk_index}"`.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| POST /v1/rag/ingest → Gateway | Document content crosses from client to gateway |
| Gateway → Vector Store | Anonymized chunk content crosses to external vector DB |
| Gateway → Audit Logger | Chunk metadata (no raw values) written to audit log |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-19-02-01 | Information Disclosure | Document content | mitigate | Document inspected in memory only. Anonymized before storage. Raw document never persists. |
| T-19-02-02 | Tampering | Vector store | mitigate | Only anonymized chunks leave the gateway. Original values stay in-memory CacheManager with TTL. |
| T-19-02-03 | Information Disclosure | Chunk metadata | mitigate | Metadata fields contain classification data, no raw entity values. entity_types_present lists types only (e.g., "email"), not values. |
| T-19-02-04 | Denial of Service | Large document | mitigate | Configurable max_document_size (default 10MB). Chunked processing limits per-chunk memory. Timeout per ingest request: 300s. |
</threat_model>
